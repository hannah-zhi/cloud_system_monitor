## 2. 工作流

1. 先确认需求影响范围：
   - 首页集中监测卡片。
   - 右侧 `预警/告警清单`。
   - 单场站 `场站概览`。
   - 单场站 `安全诊断`。
   - `安全预警 / 预警管理` 或 `安全评估`。
   - `deliverables/PRD_safety_warning_system.md`。
   - `deliverables/safety-warning-system-skill/SKILL.md`。

2. 修改代码：
   - 主要文件通常是 `app.js`、`styles.css`、`index.html`。
   - 手工文件修改使用 `apply_patch`。
   - 不要用破坏性 git 命令。
   - 不要处理无关未跟踪文件，尤其当前存在的：
     `deliverables/PRD_安全预警模块_安全评估_预警管理.md`

3. 同步文档：
   - 任何行为规则、数据规则、交互规则变化，都要同步写入 PRD 和 skill。
   - PRD 记录面向产品/需求的规则。
   - skill 记录后续复现和扩展时要遵守的实现规则。

4. 更新缓存版本：
   - 修改页面行为或样式后，必须更新 `index.html` 中 `styles.css?v=...` 和 `app.js?v=...`。
   - 版本号建议使用当天日期加语义后缀，例如：
     `20260527-offline-device-alarm-filter`
   - 如果用户反馈刷新后没看到更新，先确认线上 HTML 是否已经包含新版本；必要时再次提升版本号。

5. 本地验证：
   - 用户外接显示器看到的比例接近 Chrome 67% 缩放，涉及视觉问题时要用 67% 缩放验证。
   - Browser 插件/Node REPL 在此环境中可能出现 `拒绝访问`，可使用外部 Chrome Headless + CDP 验证。
   - 验证至少覆盖：
     - 页面是否加载。
     - 目标 DOM 是否存在。
     - 关键 class/style 是否符合预期。
     - 点击、筛选、hover、日期选择等交互是否生效。
     - 对比线上 HTML 是否引用最新缓存版本。

6. 提交发布：
   - `git diff --check`
   - `git status --short`
   - `git add` 只添加本次相关文件。
   - `git commit -m "简洁英文提交信息"`
   - `git push origin main`
   - 等 Pages 构建成功。
   - 用带 `?force=版本号` 的 URL 确认线上更新。

## 3. 重要通用规则

### 3.1 缓存与刷新

- GitHub Pages 可能缓存旧 HTML 或旧 CSS/JS。
- 每次页面样式或脚本改动后都更新 `index.html` 的资源版本号。
- 如果用户说刷新后没有变化：
  - 先用 `Invoke-WebRequest` 检查线上 HTML 是否包含新版本号。
  - 如果线上已更新，让用户打开带 `?force=版本号` 的链接或 `Ctrl + F5`。
  - 如果线上未更新，等待 Pages workflow 或重新推送版本号。

### 3.2 视觉验证比例

- 用户反馈截图通常来自外接显示器，浏览器缩放约为 67%。
- 调整布局、间距、tooltip、图表、卡片高度时，要按 67% 缩放验证。
- 不要只看 100% 缩放下的效果。

### 3.3 文件与 Git

- 保持改动范围小。
- 不要重置、回滚或清理用户未要求处理的文件。
- 当前未跟踪文件不要提交：
  `deliverables/PRD_安全预警模块_安全评估_预警管理.md`
- 每次提交后最终回复中说明：
  - 修改内容。
  - 验证方式。
  - Pages 链接。
  - commit hash。

## 4. 集中监测页面核心业务规则

### 4.1 菜单结构

- `全景监测` 是一级菜单。
- `全景概览` 和 `集中监测` 是其下二级菜单。
- `集中监测` 位于 `全景概览` 后。

### 4.2 首页场站卡片

- 默认排序：`SOS数值-从低到高`。
- 排序选项：
  - `SOS数值-从低到高`
  - `SOS数值-从高到低`
  - `场站编号-顺序`
  - `场站编号-倒序`
- 卡片边框颜色依据 SOS 风险等级。
- 右上角圆点颜色依据通讯状态：
  - 通讯正常：绿色。
  - 部分通讯中断：黄色。
  - 通讯中断：红色。
  - 离线：灰色。
- 卡片左上角状态标签优先级：
  `故障 > 告警 > 充电/放电/待机/停机`
- 离线且无设备告警/故障时，不显示左上角运行状态标签。
- 故障卡片底色为红色渐变，告警卡片底色为黄色渐变，但不得覆盖 SOS 边框颜色。
- 卡片显示：
  - SOS 进度条。
  - 健康度进度条。
  - 通讯状态。
  - 额定能量/容量。
  - 子系统数量。
  - 场站SOC。
  - 场站SOP/SOE。
- `场站SOP/SOE` 大约为额定功率/容量的 90%，格式类似 `20MW/43MWh`。
- 收起模式仅保留：
  - 左上状态标签。
  - 右上通讯圆点。
  - SOS。
  - 场站SOP/SOE。
  - 不展示 `通讯状态` 文本行。
- 筛选后卡片比例不能被拉伸，少量卡片也保持正常固定宽高。

### 4.3 首页顶部饼图筛选

集中监测首页顶部为四个饼图：

- `SOS`
- `场站预警分布`
- `场站告警分布`
- `通讯状态`

规则：

- 饼图和 legend 需要整体居中。
- legend 标签文字不能被截断，数字右对齐。
- 点击饼图区块或 legend 可以筛选下方场站卡片。
- `场站告警分布` 包含：
  - 故障
  - 告警
  - 无告警
  中心总数应等于场站总数。

## 5. 右侧预警/告警清单规则

### 5.1 首页和单站右侧清单

- 标题：`预警/告警清单`
- 来源分类为：
  - `预警`
  - `告警`
  - `数据`
- `预警` 对应云端预警。
- `告警` 对应设备告警，并派生为 `告警/故障`。
- `数据` 对应数据告警。
- 不展示 `站端预警`。
- 三个来源框均支持点击筛选下方列表。
- `预警` 框内提供 `一级/二级` 子筛选。
- `告警` 框内提供 `故障/告警` 子筛选。
- 子筛选不使用重边框按钮，使用轻量高亮或下划线。
- 子筛选标签文字使用语义色，数字保持白色。
- 数据类列表项左上角保留模块标签，如 `数据系统`，右上角显示 `数据`。
- 告警类列表项右上角统一显示 `告警`，左上角区分 `故障/告警`。
- 右侧清单过长时内部滚动，底端要和左侧内容底端对齐。

### 5.2 空态文案

单站右侧清单中，某类为 0 时显示：

- `当前场站无云端预警`
- `当前场站无设备告警`
- `当前场站无数据告警`

具体文案要跟当前筛选类别对应。

## 6. 离线场站告警规则

离线场站的正确规则是：

- 只过滤设备告警类，也就是 `告警/故障`。
- 云端预警类正常存在。
- 数据告警正常存在。

影响范围：

- 首页右侧清单：
  - 离线场站不出现设备告警类。
  - 离线场站的预警类仍展示。
  - 离线场站的数据类仍展示。
- 首页场站卡片：
  - 不因设备告警显示告警/故障底色或标签。
  - 预警类仍可影响 `场站预警分布`。
- 单站右侧清单：
  - 预警和数据正常展示。
  - 设备告警类不展示。
- 子系统卡片：
  - 状态仍显示 `离线`。
  - 设备告警类不影响卡片底色。
  - 预警类仍影响边框颜色。
- 子系统 tooltip：
  - `告警类` 为 0。
  - `预警类` 正常展示。

实现入口：

- `homeVisibleAlarm(alarm)` 排除站端预警，并调用 `alarmAllowedByStationComm(alarm)`。
- `alarmAllowedByStationComm(alarm)` 对离线场站只过滤 `homeAlarmCategory(alarm) === "alarm"`。
- 不要再把 `warning` 一起过滤掉。

## 7. 单场站概览规则

### 7.1 顶部三个指标卡

- 第一张：安全指数，使用类似 `全站当前 SOS 安全指数` 的仪表盘风格。
- 第二张：健康指数，使用护盾图标。
- 第三张：当前可用电量，使用电池图标。
- 图标在左，文字和数字在右。
- 标题不要占用额外空间，整体比例要和参考图一致。

### 7.2 场站运行 / 场站属性 / 充放电统计

- 三个卡片高度、外边框比例应保持一致，不被右侧清单拉伸。
- 右侧清单数量多时只让右侧列表内部滚动。
- `场站运行` 使用半圆仪表盘，0%/100% 不得遮挡图形。
- `场站属性` 包含：
  - 场站类型。
  - 系统数量。
  - 额定容量。
  - 额定功率。
- `充放电统计` 替代原 `储能系统`。
- 右上角有 `当日 / 当月 / 当年` 切换。
- 内容使用两个水波纹图标展示充电量、放电量。

### 7.3 拓扑图 / 子系统卡片

- 子系统命名格式：`项目号-#序号子系统`，例如 `K0060-#1子系统`。
- 子系统数量必须与场站系统数量一致。
- 一排最多 6 个子系统。
- 子系统少于一排时整体居中。
- 多排时线束从左上总线连接到每排总线，再连接到每个子系统。
- 子系统卡片右上角状态标签使用：
  - 充电
  - 放电
  - 待机
  - 停机
  - 离线
  - 告警
  - 故障
- 状态优先级：
  `故障 > 告警 > 离线 > 充电/放电/待机/停机`
- 设备告警/故障控制卡片底色。
- 预警等级控制卡片边框。
- 同一个子系统既有告警又有一级预警时，应是告警底色 + 一级预警边框。

### 7.4 子系统 tooltip

- tooltip 分区顺序：
  1. `告警类`
  2. `预警类`
  3. 如有需要再显示 `数据类`
- `告警类：1` 等标题和圆点使用对应语义色。
- 具体告警/预警名称保持白色。
- 无告警时显示：`当前子系统暂无告警`
- 无预警时显示：`当前子系统暂无预警`
- 即使告警和预警都为 0，也要分成两段展示，不要合并成 `当前子系统暂无预警/告警`。
- 无告警/无预警 tooltip 的格式要匹配有内容 tooltip 的下半段：
  - `告警类 / 预警类` 标题左对齐。
  - 空态文案居中。
  - 分割线宽度与内容区一致。
  - 标题、文案、分割线之间距离与有内容 tooltip 一致。
  - 不出现无用滚动条。
